## 三件使用者沒問、但對這個環境最重要的事

### 1. 業務數據不在這個 repo 裡——在使用者的瀏覽器裡
這是本環境最大的單點風險。倉存系統的產品、訂單、發票、批次數據存在 localStorage，
**換機、清快取、瀏覽器故障 = 業務數據消失**。制度寫得再好也救不了這個。
所以：
- 任何 session 有機會就推進 Supabase 雲端同步的實際啟用（SQL 在 `supabase/schema.sql`，步驟在 `inventory/README.md`）。使用者已同意方向，只差實際配置。
- **任何改動不得破壞匯出/匯入相容性**（`store.js` 的 `exportAll`/`importAll`）。改 schema 要寫遷移邏輯，舊備份檔必須仍可匯入。
- 提醒使用者定期匯出備份不是嘮叨，是這個架構下唯一的災難恢復。

### 2. 這個 repo 是公開部署的（GitHub Pages）
所有 commit 內容視同公開，包括這些制度檔（無妨，裡面沒祕密——保持這樣）。
推論：
- 永遠不把 API key、密碼、客戶資料寫進 repo。Supabase 憑證的正確位置是設定頁→localStorage。
- 倉存系統的代碼任何人看得到：安全必須靠 Supabase RLS（伺服器端），不能靠前端遮蔽。
  前端的標籤密碼閘只是操作性遮蔽，這一點已明寫在 `inventory/README.md`，不要在對話裡誇大它。

### 3. 使用者是營運者，不是開發者
訊號：粵語溝通、需求圍繞表單/打印/針機/月結/回篩。這決定交付標準：
- 回覆用粵語/繁中，先講結果後講細節，短。
- UI 字串一律繁中；「可以用的表單」勝過「優雅的 API」。
- 打印類功能（送貨單、標籤）的最終驗收在**實體打印機**，harness 驗不到——明說，並請使用者實印一張回報。
- 使用者可能從手機發訊息、可能中途斷線：不要把關鍵資訊只放在長對話中段，落檔 + 收尾總結。

## 這套制度最可能的退化方式（與預防）

### 退化 1：把制度當儀式
症狀：口稱「遵守 CLAUDE.md」但沒讀對應檔、沒寫驗收條件、沒驗證。
預防：任務開始時講明「本任務讀了哪份制度檔」；收尾必列 verification；沒驗證就寫「沒驗證」。

### 退化 2：能力清單過期而無人更新
症狀：dispatch 檔裡的模型名/工具名與實際 harness 脫節，弱模型照抄呼叫失敗，然後**繞過制度自由發揮**。
（這正是 2026-07-06 Codex 制度包的死法——整包規則綁死在 `gpt-5.*` 上。）
預防：呼叫失敗的第一反應不是放棄制度，是按 `04-maintenance-protocol.md` 第 3 節更新
`01-model-dispatch.md` 第 0 節，然後繼續照制度走。

### 退化 3：小模型自信過度
症狀：路徑、工具名、API 行為靠記憶答；失敗後同路重試；高風險任務不升級。
預防：`02-judgment-rubrics.md` 1.1（事實錯一次即升）與 4.1（兩敗換路）是硬規則不是建議。

### 退化 4：CLAUDE.md 膨脹
症狀：每個教訓都塞進 CLAUDE.md，啟動 token 越吃越多，弱模型讀完反而不知道做什麼。
預防：CLAUDE.md 只做路由 + 核心規則；教訓按維護協議第 5 節寫到對應檔。

### 退化 5：完成定義偷換
症狀：「寫了」當「完成」；「看起來對」當「驗證」；「工具跑不了」被靜默吞掉。
預防：CLAUDE.md 完成閘四條件缺一不可；跑不了的驗證寫明缺什麼（正例：Playwright 缺件時
明寫「視覺驗證缺口」而不是跳過）。

### 退化 6：制度檔和專案檔互相污染
症狀：制度檔裡越來越多倉存 app 細節，app 文檔裡出現調度規則，兩邊都難維護。
預防：專案事實 → `inventory/README.md` / `COMPARISON.md`；制度 → `.claude/institution/`。
維護協議第 5 節的路由表是仲裁標準。
