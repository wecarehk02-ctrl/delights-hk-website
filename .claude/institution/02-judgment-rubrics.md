# D. 判斷力外化：弱模型可執行的 Rubric

建立：2026-07-07。用法：拿不準時**不要即興發揮**，找到對應 rubric 照 trigger→action 執行。
每條有 Trigger（何時適用）、Action（做什麼）、一個正例、一個反例。正例多數來自本 workspace 真實事件。

---

## 1. 何時該升級模型

### 1.1 關鍵事實錯一次即升
Trigger：模型講錯了檔名、工具名、模型名、命令、API 或需求，且這個錯會影響下一步。
Action：停 → 重查事實源 → 升一級（haiku→sonnet→opus）→ 把錯誤寫進新 prompt 讓升級後的模型不重犯。
- 正例：發現制度檔裡的 `multi_agent_v1.spawn_agent` 在本環境不存在 → 不沿用，改查當前 Agent 工具 schema，並修正制度檔。
- 反例：「工具名大概差不多」→ 繼續用錯名寫規則（zip 制度包就是這樣過期的）。

### 1.2 高風險域強制升級驗收
Trigger：任務碰到發票金額、階梯價、庫存數量、刪除、auth、對外發送、客戶資料、migration。
Action：實作至少 sonnet；交付前必須有 fresh-context 驗收（opus 審查或實跑測試）；沒驗到的寫明。
- 正例：改階梯價計算 → sonnet 實作 → 用實際數字算三組（0%/5%/10% 檔位邊界）比對 → `/code-review` 收尾。
- 反例：haiku 順手改了 `tierPrice()` 的四捨五入，主對話說「看起來對」就交付。

### 1.3 品味題先轉判準
Trigger：需求含「更好、更專業、更高級、cleaner」等品味詞。
Action：先把品味轉成可見判準再動手；轉不出來（仍主觀）→ 問使用者或標明假設。**這是 harness 的真極限**（見第 6 節），不要假裝流程能補。
- 正例：「發票要專業」→ 轉成：對齊的金額欄、A4 可印、無截字、公司抬頭齊、行項可編輯。
- 反例：加一堆漸變色然後宣稱「更專業了」。

## 2. 何時算真的完成

### 2.1 產物存在（最低門檻）
Trigger：任何建檔/改檔任務。
Action：用 filesystem 讀取確認存在；制度檔要 read-back 標題結構。工具回報成功 ≠ 你驗過。
- 正例：寫完 6 份制度檔後 `grep -c '^#' 每份` + 確認路由檔連結指向存在的路徑。
- 反例：「已建立」——因為 Write 沒報錯，但從未列出檔案。

### 2.2 行為已驗證（按驗證階梯）
Trigger：任何聲稱「軟件行為改變了」的任務。
Action：跑可用的最強驗證：實跑/e2e > 單元邏輯驗證 > `node --check`。跑不了強的，寫明**確切原因**與缺口。驗證訊號要過濾噪音（本 repo：CDN 封鎖噪音見 CLAUDE.md）。
- 正例（本 session 真事）：QR 結構檢查全過後仍做窮舉 round-trip 解碼，在 v9 抓到 alignment bug——結構對≠掃得到。
- 反例：「`node --check` 過了，功能應該正常」——語法檢查驗不出任何行為。

### 2.3 下一個 session 能從磁碟接手
Trigger：產物型任務、政策建立、交接、或可能被中斷的長任務。
Action：最終回覆前，把狀態、決定、未完成項寫入檔案。純問答與無檔審查豁免（答案本身就是產物）。
- 正例：context 將盡 → 停止新產出，把未完成項寫進 `05-letter-to-future-sessions.md` 再收尾。
- 反例：重要的 next steps 只在聊天最終回覆出現，session 一關就蒸發。

## 3. 何時該停下來問使用者

### 3.1 不可逆對外動作
Trigger：發真實 email/webhook、公開發佈、刪數據、扣款、改生產設定、暴露私人資料。
Action：除非使用者**本次**明確授權該類動作，否則停下報告。注意 `AskUserQuestion` 可能失敗（實證）——問不到就不做並在產出裡標明，**不可逆動作寧可漏做不可錯做**。
- 正例：補貨 email 停在「生成草稿給人手發送」；真正自動發送需使用者配置 webhook（等於授權）。
- 反例：因為功能存在就把月結發票直接發給真實客戶。

### 3.2 業務政策未知
Trigger：多個合理業務選項，選錯會造成營運痛苦（例：庫存在下單時扣還是出貨時扣）。
Action：能問就問一條精準問題；使用者要求自主時，選**可逆**預設並在產出裡標明假設。
- 正例（本 repo 現狀）：庫存在訂單轉「已出貨」時扣（FIFO 按到期日），此決定寫在 `inventory/js/biz.js` 的 `deductStock`——改動前先確認使用者意圖。
- 反例：不聲不響把扣貨時機改掉，發票照舊，帳實不符。

### 3.3 缺密鑰/憑證
Trigger：需要 API key、資料庫 URL、SMTP 等。
Action：不編造值；建 placeholder + 寫清楚去哪裡填（本 repo 慣例：設定頁輸入、存 localStorage、絕不入 repo——repo 是公開部署的）。
- 正例：Supabase 整合做成「設定頁填 URL+anon key」，repo 裡零憑證。
- 反例：把測試用 endpoint 寫死進代碼並宣稱「自動化完成」。

## 4. 什麼訊號代表方向錯了（換路，不是重試）

### 4.1 同法兩敗
Trigger：同一命令/測試/思路失敗兩次，結果相似。
Action：禁止第三次。選：最小重現 / 檢查底層假設 / 升級 / fresh verifier / 問人。
- 正例（本 session 真事）：自寫的 RS syndrome 檢查兩種約定都非零 → 不再試第三種約定，改用定義性的多項式除法驗證 → 立即分清是測試錯不是編碼器錯。
- 反例：把同一個失敗的 syndrome 公式再跑三次，期望結果不同。

### 4.2 證據與計劃矛盾
Trigger：檔案結構、工具可用性、API 行為與計劃預設不符。
Action：更新計劃 + 寫下新假設；不要硬推原路線。**警訊詞**：發現自己在想「照理應該可以」。
- 正例：repo 沒有 CLAUDE.md → 直接建新檔，不假裝備份了不存在的檔。
- 反例：報告「已備份 CLAUDE.md」但該檔從未存在過。

### 4.3 產出變寬但驗收很窄
Trigger：開始加無關功能、大段解釋、超範圍重構。
Action：重讀驗收條件 → 砍無關工作 → 有價值的額外想法寫進「future work」一段，不是做出來。
- 正例：制度任務就寫制度檔；發現 app 有可改進處 → 寫進 COMPARISON.md 的移植清單，不動手。
- 反例：本該寫規則的 session 跑去重寫 app CSS，因為「順手」。

## 5. 品質底線怎麼驗

### 5.1 制度/文件檔底線
逐項檢查：目的明確｜trigger 和 action 具體｜無「無範例的抽象要求」｜路徑存在（或標明計劃中並事後驗證）｜不與 CLAUDE.md 打架｜弱模型讀得懂照著做。
- 正例：「輸出超過 80 行→落檔，回路徑+5 條摘要」。
- 反例：「保持簡潔和高品質」。

### 5.2 程式碼底線
逐項檢查：最小合理範圍｜跟隨現有風格（本 repo：ES5 風格 IIFE 模組、繁中 UI 字串）｜無無關重構｜錯誤路徑考慮過｜跑過檢查｜UI 改動有實際看過（e2e 或截圖）。
- 正例：加一個驗證 helper，風格與鄰近代碼一致，跑了 e2e。
- 反例：修一個 bug 順便重寫整個模組因為「覺得亂」。

### 5.3 研究/事實類底線
用一手或當前來源｜引用來源或工具輸出｜事實與推論分開標｜未解決的不確定性標明。
- 正例：「Agent 工具 model enum 實測為 haiku/sonnet/opus/fable（來源：本 session tool schema）」。
- 反例：「用 Sonnet 4.5 high effort」——沒查過這個名字在本環境是否存在。

## 6. Harness 極限與誠實條款

流程（拆解、read-back、測試、fresh review、多答案評審）補得了**執行品質**。
流程補不了：
- 品味與模糊判斷（什麼叫「高級感」）
- 使用者沒講的業務政策與價值取捨
- 不在場的外部系統（真實針機、真實 email 收發、Supabase 生產行為）
- 私人上下文（使用者腦裡的計劃）

遇到極限時的標準回覆格式：
```text
我能驗證 X（方法：…）。
我在此環境無法驗證 Y，因為 Z。
最佳下一步是 A。
若屬主觀取捨，請使用者在 B / C 之間選。
```
- 正例：「檔案存在且語法過。真實 Epson 打印我無法驗證（無實體打印機），下一步是使用者實印一張並回報邊界裁切情況。」
- 反例：「打印功能已完整整合」——實際只驗過瀏覽器列印對話框會彈出。
